Architecture Overview

How the pieces fit together

Three systems work together. Sanity owns content and imagery. Stripe owns money, discount codes, and payment sessions. The Next.js app is the glue — it reads from both and writes back to Sanity via webhooks after a purchase completes.

Content Layer

Sanity CMS

  • Product images & descriptions
  • Rich text / story content
  • Stock quantity (number field)
  • In-stock boolean flag
  • Stripe Product ID reference

Application Layer

Next.js 15

  • CartContext + localStorage
  • CartDrawer UI component
  • /api/checkout — creates Stripe session
  • /api/webhooks/stripe — listens for events
  • Shop, product detail, success pages

Commerce Layer

Stripe

  • Products & Prices (source of truth for pricing)
  • Hosted Checkout page
  • Promo Codes / Coupons
  • Webhooks on checkout.session.completed
  • Order history in Stripe Dashboard

On inventory: You said Sanity cannot control quantity — meaning you cannot let customers change stock through Sanity's hosted UI. The fix: stock is stored as a number field in Sanity, but it is only ever written to programmatically via the Stripe webhook after a real purchase. You never rely on a Sanity editor manually updating it. This is how professional stores handle it.

Phase 01

The Cart System

The cart lives entirely on the client — no database, no server round-trips. State is managed with React Context and mirrored to localStorage so it survives page refreshes and browser restarts. A slide-out drawer (animated with GSAP, matching your existing animation style) opens from the right on any "Add to Cart" action.

What gets built

  • CartContext with useReducer — actions for add, remove, update quantity, clear
  • useLocalStorage hook that syncs cart state automatically on every change
  • CartDrawer component: slides in from the right, lists items with image, name, price, and quantity stepper
  • Cart icon added to the Navbar with an animated item-count badge
  • Subtotal display and a "Proceed to Checkout" button at the bottom of the drawer
  • Quantity controls in the drawer — plus/minus buttons, remove item button
  • Empty cart state with a friendly message and link to the shop

Cart state shape

// lib/cartTypes.ts type CartItem = { id: string; // Stripe Price ID name: string; price: number; // in öre / cents currency: string; image: string; quantity: number; slug: string; }; type CartState = { items: CartItem[]; isOpen: boolean; };

Quantity in the cart is separate from inventory. A customer can add 3 of the same piece to their cart — the actual stock check happens server-side at checkout creation, not in the browser. This prevents race conditions and overselling.

Phase 02

Product Shop Infrastructure

Sanity already holds your beautiful product content. We add a handful of fields to the existing schema, create a /en/shop and /sv/shop page, and wire up "Add to Cart" buttons. Pricing lives in Stripe — Sanity only stores the Stripe Price ID as a reference, so you never manage a price in two places.

Sanity schema additions

  • stripePriceId (string) — the Stripe Price ID like price_1Abc…. This is what gets sent to Stripe at checkout.
  • price (number) — display price in SEK/EUR, fetched from Stripe on first render and cached. Shown in the UI.
  • currency (string) — "sek" or "eur"
  • stockQuantity (number) — total units available. Set this manually when you add a product. Decremented automatically after each purchase via webhook.
  • inStock (boolean) — auto-set to false when stockQuantity reaches 0. Used to show "Sold Out" on the product card and disable the cart button.
  • availableForPurchase (boolean) — a manual override you can flip to hide a product from the shop without deleting it.

New pages

  • /[locale]/shop — the main shop grid page. Fetches all products from Sanity where availableForPurchase is true. Uses your existing PortfolioGrid-style layout.
  • /[locale]/shop/[slug] — individual product detail page with full imagery, description in Portable Text, price, stock status, and "Add to Cart" button.
  • /[locale]/checkout/success — post-payment confirmation page shown after Stripe redirects the customer back. Displays order summary from Stripe session params.
  • /[locale]/checkout/cancel — shown if the customer cancels on the Stripe page. Reassures them their cart is still intact and links back to the shop.

Nav addition: A "Shop" link will be added to the Navbar's navLinks array in both locales — "Shop" in English, "Butik" in Swedish.

Phase 03

Stripe Checkout Integration

When the customer clicks "Proceed to Checkout", the browser sends the cart to a Next.js API route. That route validates stock server-side, then creates a Stripe Checkout Session and returns its URL. The browser then redirects there. Stripe handles the entire payment — card entry, 3D Secure, Apple Pay, Google Pay — and redirects back to your success page when done.

API route: /api/checkout

  • Receives the cart as JSON in the request body
  • For each item, queries Sanity to verify stockQuantity is sufficient — if any item is out of stock, returns a 400 error with a clear message so the UI can inform the customer
  • Builds Stripe line_items from each cart item's stripePriceId and quantity
  • Creates a Checkout Session with mode: "payment"
  • Sets success_url to /checkout/success with session_id in the query string
  • Sets cancel_url to /checkout/cancel
  • Passes locale and a cart snapshot as session metadata (used in the webhook)
  • Returns the session URL — the browser redirects to it

Checkout session config

// app/api/checkout/route.ts await stripe.checkout.sessions.create({ mode: 'payment', line_items: cartItems, allow_promotion_codes: true, // ↑ Enables your Stripe promo // codes natively. No extra code. shipping_address_collection: { allowed_countries: ['SE', 'NO', 'DK', 'FI', 'DE', 'GB'], }, phone_number_collection: { enabled: true }, customer_email: undefined, // pre-fill if user is logged in success_url: `${BASE_URL}/checkout/success ?session_id={CHECKOUT_SESSION_ID}`, cancel_url: `${BASE_URL}/checkout/cancel`, metadata: { locale: locale }, });

Payment methods: Stripe Checkout automatically shows Apple Pay, Google Pay, and card entry — no extra configuration needed. It also handles currency, tax display, and accessibility out of the box. This is why hosted checkout is the right call for a premium site: the payment experience is already world-class.

Phase 04

Post-Purchase Flow

After Stripe processes the payment, it fires a webhook to your server. That webhook handler does three things: sends the customer an order confirmation email, decrements stock in Sanity, and marks products as out of stock if they hit zero. This is the backbone of a professional storefront — fully automated, zero manual effort.

Webhook handler: /api/webhooks/stripe

  • Verifies the Stripe signature using STRIPE_WEBHOOK_SECRET to ensure the request is genuine and not spoofed
  • Listens for checkout.session.completed — the most important event, fired the moment payment succeeds
  • Extracts the line items and customer details from the session
  • Calls Resend to send an order confirmation email to the customer
  • For each purchased product, calls the Sanity write API to decrement stockQuantity by the quantity purchased
  • If stockQuantity reaches 0 or below, also flips inStock to false on the same document
  • Returns 200 quickly — Stripe expects a fast response and will retry if it times out

Order confirmation email

  • Sent via Resend — the cleanest transactional email solution for Next.js, with a generous free tier (3,000/month) and excellent deliverability
  • Template uses React Email components styled to match the Wallens Design brand: cream background, forest headings, gold accents
  • Contains: order number, list of items with images and prices, subtotal, shipping address, and a note about the handcrafted nature of the pieces
  • Sent from a custom domain address like orders@wallensdesign.se for professionalism
  • You also receive a copy (BCC) at your own address so you always know when an order comes in

Why Resend over SendGrid/Mailgun? Resend is built for Next.js, uses React for email templates (same codebase), and has a dead-simple API. SendGrid is powerful but has a steep setup curve for relatively simple transactional needs.

Phase 05

Discount Codes & Promotions

Discount codes are entirely managed in the Stripe Dashboard — you create them there with whatever rules you want (percentage off, fixed amount, minimum order value, single use, expiry date). Because the checkout session is created with allow_promotion_codes: true, Stripe shows a promo code field on the checkout page automatically. Zero extra frontend code needed.

What you can do in Stripe Dashboard

  • Create percentage-off codes (e.g. WELCOME10 for 10% off)
  • Create fixed-amount codes (e.g. SUMMER50 for 50 SEK off)
  • Set minimum cart value requirements
  • Limit codes to a single use per customer, or a total number of redemptions
  • Set expiry dates on codes
  • Restrict codes to specific products or categories
  • See redemption analytics in the Stripe Dashboard

How it appears to the customer

  • On the Stripe Checkout page there is a "Have a promo code?" link or field
  • The customer enters their code, Stripe validates it instantly, and the discount is applied to the order total before payment
  • Invalid or expired codes show a clear error from Stripe — no custom error handling needed on your end
  • The discount amount is visible on the checkout page and on the order confirmation receipt Stripe sends automatically
  • Your order confirmation email from Resend can also display the discount applied

Phase 06

Inventory Management

Since Sanity's UI cannot drive inventory (you cannot have customers write to Sanity), the flow is inverted. Sanity stores the numbers, but only your webhook ever writes to them. You set the initial stock count when adding a product. Everything after that is automatic.

Inventory flow

  • You add a product — set stockQuantity to e.g. 1 (for a one-of-a-kind piece) or 5 for a small run. Set inStock to true.
  • Customer adds to cart — no stock check yet, cart is purely client-side
  • Customer proceeds to checkout — the /api/checkout route reads Sanity stock. If any item has stockQuantity less than the requested quantity, the request fails with a friendly error before Stripe is ever called
  • Payment completes — Stripe webhook fires. Handler decrements stockQuantity by quantity purchased using the Sanity PATCH API
  • Stock reaches zero — webhook also sets inStock: false on the product document
  • Product card updates — next time the shop page renders, inStock: false causes the card to show "Sold Out" and disable "Add to Cart"

Stock update in webhook

// After payment.succeeded await sanityClient .patch(productId) .dec({ stockQuantity: qtySold }) .commit(); // Then check if now zero const updated = await sanityClient .fetch(`*[_id == $id][0].stockQuantity`, { id: productId }); if (updated <= 0) { await sanityClient .patch(productId) .set({ inStock: false }) .commit(); }

For bespoke / made-to-order pieces: Set stockQuantity to 1. When it sells, it becomes "Sold Out" automatically. If you want to re-list it, you flip inStock back to true and reset the count in Sanity Studio. Takes 10 seconds.

Configuration

Environment variables

All secrets go in .env.local (already gitignored). The NEXT_PUBLIC_ prefixed ones are safe to expose to the browser — they are public keys only. Everything else is server-only.

Variable Where to find it Notes
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY Stripe Dashboard → Developers → API keys Public key, safe in the browser. Use the test key (pk_test_…) while developing.
STRIPE_SECRET_KEY Stripe Dashboard → Developers → API keys Server-only. Never expose this to the client. Use sk_test_… while developing.
STRIPE_WEBHOOK_SECRET Stripe Dashboard → Developers → Webhooks → Signing secret Generated after you create the webhook endpoint. Used to verify webhook authenticity. For local testing, use the Stripe CLI to get a local webhook secret.
RESEND_API_KEY resend.com → API Keys Server-only. Free tier covers 3,000 emails/month. Add your sending domain in the Resend dashboard first.
NEXT_PUBLIC_SITE_URL Your own config e.g. https://wallensdesign.se in production, http://localhost:3000 in development. Used for redirect URLs in Stripe sessions.
SANITY_API_TOKEN Sanity Manage → Project → API → Tokens Must have Editor or higher permissions so the webhook can write stock updates back to Sanity. Keep server-only.

Dependencies

Packages to install

Three new packages. That is it. No bloated e-commerce frameworks. The cart is pure React state, Stripe handles payments, and Resend handles email.

stripe

Stripe's official Node.js SDK. Used server-side only in API routes and the webhook handler. Handles session creation and webhook signature verification.

@stripe/stripe-js

Stripe's browser library. Loaded lazily (only on checkout). Used if you ever add Stripe Elements later. Also handles the redirect to Stripe Checkout.

resend

Minimal transactional email client. Pairs perfectly with React Email for branded templates. Used in the webhook handler after checkout.session.completed fires.

react-email

Optional but recommended. Lets you write the order confirmation email as a React component using the same design patterns as the rest of the site.

# Install all four at once npm install stripe @stripe/stripe-js resend @react-email/components

Timeline

Suggested build order

Each phase unlocks the next. Do not skip ahead — a cart without a checkout route is not testable, and a checkout without a webhook means stock never decrements.

Week 1

Cart system + Sanity schema

Build CartContext, useLocalStorage, CartDrawer, CartIcon. Update Navbar. Add the new Sanity fields and set stock on a few test products. At the end of this phase the cart is fully functional in the browser, items persist across refreshes, and you can add/remove/adjust quantity from the drawer.

Week 2

Shop pages + Add to Cart

Build the /shop listing page and /shop/[slug] product detail page. Wire up AddToCartButton to CartContext. Add Shop to the nav. By the end of this phase the full browse-to-cart flow works end to end in the UI.

Week 3

Stripe Checkout + success/cancel pages

Build /api/checkout with stock validation. Create success and cancel pages. Test the full purchase flow in Stripe test mode using the test card 4242 4242 4242 4242. Add promo code tests. Verify the redirect back to your site works correctly.

Week 4

Webhooks + Email + Inventory

Build /api/webhooks/stripe. Test locally with the Stripe CLI (stripe listen --forward-to localhost:3000/api/webhooks/stripe). Implement Resend email sending. Verify stock decrements in Sanity after a test purchase. Deploy to production and register the live webhook URL in the Stripe Dashboard.

Before launch

Switch to live Stripe keys + final QA

Swap pk_test_ and sk_test_ for live keys in production environment variables. Create real products in Stripe. Create a real promo code. Do a live test purchase with a real card. Verify the confirmation email arrives, stock decrements, and the success page shows correctly.
